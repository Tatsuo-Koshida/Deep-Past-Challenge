# 目的（あなたへの依頼）

これから送るテキストとスクリーンショット画像は、Kaggle コンペ **Deep Past Challenge - Translate Akkadian to English** の学習データ候補です。  
1 件の長いレコードについて、**アッカド語の転写（transliteration）** と **英語訳（translation）** の対応を崩さないように注意しながら、**複数の短い独立レコードへ分割**してください。

この作業の目的は、ByT5 モデルで扱いやすいように、各レコードを **1024 byte 未満**に収めることです。  
ただし、**無理に切って対応関係を壊すくらいなら、分割不能として返す**ほうを優先してください。

---

## 背景（最小限）

- このコンペでは、アッカド語の転写から英語訳を生成するモデルを学習します。
- ByT5 は byte-level のモデルなので、文字数ではなく **UTF-8 byte 長**が効きます。
- source 側では通常 `translate Akkadian to English: ` という prefix が付きます。
- 実務上、1 レコードあたり source / target ともに **1024 byte 未満**に抑えたいです。
- 正確な byte 数をあなたが毎回厳密計算する必要はありません。**安全側に倒して短めに切る**ことを優先してください。

---

## あなたにやってほしいこと

1. 与えられた `transliteration` と `translation` を読む
2. 必要ならスクリーンショット画像も参照する
3. **意味的に対応している境界**を見つける
4. 1 件を複数の `transliteration_chunk` / `translation_chunk` に分ける
5. 分割が危険なら `needs_manual_review` と返す

---

## 絶対ルール

### 1) これは「分割」であり「修正」ではありません

- `transliteration` と `translation` の本文は、**原則として入力テキストの部分文字列をそのまま使ってください**
- 勝手な補完、意訳、要約、翻訳し直しは禁止です
- OCRミスや不自然な英語が見えても、**このタスクでは直さない**でください

### 2) 順序を保存してください

- 分割後のチャンクを上から順に並べると、元の `transliteration` / `translation` の順序が維持されている必要があります

### 3) 対応が見える境界でしか切らないでください

- 英語側の文や節の区切り
- 品目列挙の区切り
- 書簡定型句の切れ目
- 命令文・条件文の切れ目
- 転写側の明確な行まとまり

このような **両側でまとまりが見える場所**だけで切ってください。

### 4) 次のものは途中で切らないでください

- 数値、分数、小数、税率
- 銀量や shekel / mina のまとまり
- 人名、神名、地名
- `son of ...` や `belonging to ...` のまとまり
- 引用符の内側の途中
- 品目名の途中

### 5) 無理なら保留してください

次のような場合は、無理に分割せず `needs_manual_review` を返してください。

- 画像を見ても対応境界が確定できない
- train の英訳が途中で壊れているように見える
- 原文と英訳がずれていて 1:1 対応が不明
- どう切っても数値列や固有名詞列を破壊してしまう

---

## 画像の使い方

スクリーンショット画像は、**どこで切ると transliteration と translation の対応が自然か**を判断するために使ってください。

重要:

- PDFに写っているアッカド語と英訳文は、チャットで直接渡される `transliteration` / `translation` のテキストと**表記が一致しないことがあります**
- これは `<gap>` のような置き換え、記号整理、その他の整形が、チャットで渡すテキスト側に入っている可能性があるためです
- したがって、**PDFは対応境界を見つけるための参考資料**として使い、最終的な出力本文は **チャットで直接渡されたテキストを優先**してください
- PDFの見た目に合わせて、チャットで渡された本文を別表記へ書き換えないでください

画像を使うときの優先順位:

1. `transliteration` と `translation` の対応境界を確認する
2. train テキストだけでは曖昧な切れ目を補う
3. どこからどこまでが同じ意味のまとまりかを判断する

画像を使っても **新しい翻訳文を作る**ことはしないでください。

---

## コンペ文脈の注意点

### transliteration 側

- アッカド語転写にはハイフン、特殊文字、下付き数字、決定詞 `{d}` `{ki}` などが含まれます
- `<gap>` などの欠損表現も意味のある文字列として扱ってください
- これらを消したり正規化したりせず、**入力どおり維持**してください

### translation 側

- 英訳は OCR / LLM 補助由来で、不自然な句読点や引用符崩れが含まれることがあります
- そのため、**英語としてのきれいさ**より **対応関係の維持**を優先してください

---

## どこで切るとよいかの目安

### 切りやすい場所

- `a-na ... qí-bi-ma um-ma ...` のような導入句の後
- `if ...`, `then ...`, `thereof ...`, `all this ...` など英訳側のまとまり
- 列挙項目が切り替わる場所
- 同じ構文の反復が終わる場所

### 切りにくい場所

- 金額や数量の途中
- 固有名詞の途中
- 引用文の途中
- 1つの品目説明の途中

## 出力形式（厳守）

**JSON 配列だけ**を返してください。余計な説明文は不要です。

### 分割できる場合

```json
[
  {
    "record_id": "元レコードID",
    "split_id": "元レコードID__01",
    "status": "ok",
    "transliteration_chunk": "ここに分割後の転写本文",
    "translation_chunk": "ここに対応する英訳本文",
    "boundary_reason": "なぜこの境界で切ったかを短く説明",
    "confidence": "high"
  },
  {
    "record_id": "元レコードID",
    "split_id": "元レコードID__02",
    "status": "ok",
    "transliteration_chunk": "ここに分割後の転写本文",
    "translation_chunk": "ここに対応する英訳本文",
    "boundary_reason": "なぜこの境界で切ったかを短く説明",
    "confidence": "medium"
  }
]
```

### 分割が危険な場合

```json
[
  {
    "record_id": "元レコードID",
    "split_id": "元レコードID__00",
    "status": "needs_manual_review",
    "transliteration_chunk": "",
    "translation_chunk": "",
    "boundary_reason": "対応境界を安全に決められないため",
    "confidence": "low"
  }
]
```

---

## `confidence` の目安

- `high`: 画像とテキストの両方から、対応境界がかなり明確
- `medium`: ほぼ妥当だが、片側の対応が少し弱い
- `low`: train 原文の欠落やズレが疑われる

---

## 実際の依頼を受けたら守ること

- JSON 配列以外を返さない
- 分割であって修正ではないことを忘れない
- 画像を見ても、本文にない語を補わない
- 迷ったら `needs_manual_review` を返す

---

## 入力の例

以下のような情報が与えられます。

```md
record_id: 12345

transliteration:
...原文...

translation:
...英訳...

screenshot notes:
- image_1 は転写上段
- image_2 は英訳下段
- 同一ページ
```

このとき、あなたは上記ルールに従って JSON 配列だけを返してください。
